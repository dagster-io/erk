## Prevention Insights

Errors and failed approaches discovered during implementation:

### 1. Documentation drift from verbatim code

**What happened:** The automated `audit-pr-docs` review flagged 3 instances of verbatim code copied into learned docs across 3 files. One was an inline code snippet (`has_blocking_indicators = any(...)`), one was a code block with implementation details, and one referenced a private method name.

**Root cause:** Agents copying source code directly into documentation prose without recognizing the maintenance burden. Code in documentation goes stale silently when source files change.

**Prevention:** Use source pointer HTML comments for navigation combined with conceptual prose descriptions. Never copy code blocks or inline snippets. The pointer enables finding the code; the prose explains what it does without coupling to implementation details.

**Recommendation:** TRIPWIRE

### 2. Private method name brittleness

**What happened:** Documentation referenced `_parse_pr_from_timeline_event()` by exact name in narrative prose.

**Root cause:** Treating private method names as stable references. Private methods are internal implementation details that change frequently during refactoring - they get renamed, extracted, inlined, or deleted.

**Prevention:** Document what the code does conceptually, not what it's called. Use domain language ("the timeline event parser") rather than implementation identifiers.

**Recommendation:** TRIPWIRE

### 3. Silent command completion confusion

**What happened:** The `erk exec update-pr-description` command returned empty output, making it unclear if it succeeded or failed.

**Root cause:** Command executed successfully but produced no output.

**Prevention:** Commands should always output success/failure JSON or a clear status message. Silent completion is ambiguous.

**Recommendation:** CONTEXT_ONLY (not documentation-worthy, but worth noting for CLI design)
