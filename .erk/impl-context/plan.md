# Fix: Double activation when `erk pr checkout --script --sync` triggers direnv

## Context

When running `source "$(erk pr checkout X --script --sync)"`, the activation (venv setup, `uv pip install`, .env loading) runs **twice**:

1. The temp script does `cd /worktree` ‚Üí triggers direnv's `chpwd` hook ‚Üí direnv sources `.envrc` ‚Üí `.envrc` calls `source_env_if_exists .erk/bin/activate.sh` ‚Üí **full activation**
2. The temp script continues ‚Üí runs its own activation code (same venv creation, `uv pip install`, .env loading) ‚Üí **duplicate activation**

Both the temp script and `.erk/bin/activate.sh` are generated by `render_activation_script()`, so they contain identical activation logic. The `cd` in the temp script synchronously triggers direnv (via zsh's `chpwd_functions`), which already does the activation before the temp script's own activation section runs.

This affects both manual `source "$(erk pr checkout ...)"` and the TUI's cmux sync workflow (which passes this command via `cmux new-workspace --command`).

## Fix

Add a `$VIRTUAL_ENV` guard in `render_activation_script()` around the activation section. After `cd`, if `$VIRTUAL_ENV` already points to this worktree's `.venv`, skip activation (direnv already did it).

### File: `src/erk/cli/activation.py` ‚Äî `render_activation_script()`

Wrap the activation block (from `unset VIRTUAL_ENV` through shell completion) in:

```sh
if [ "$VIRTUAL_ENV" != "{worktree_path}/.venv" ]; then
  # ... existing activation code ...
fi
```

Keep `post_cd_commands` and `final_message` **outside** the guard ‚Äî they must always run.

Shell completion (`eval "$(erk completion ...)"`) stays **inside** the guard ‚Äî direnv's subshell can't set up shell functions, but when the activation runs once (either from direnv or from the script), completion gets set up in the current shell. When the guard skips (because direnv already ran activate.sh), completion was already set up by the activate.sh sourced via direnv, but only as env diffs, not functions. However, `erk completion` output is just function definitions that are idempotent and fast ‚Äî the cost of double-running is negligible compared to `uv pip install`. The expensive operations (uv pip install, venv creation) are the priority to guard.

**Revised approach**: Keep completion inside the guard for simplicity. The primary benefit is avoiding the duplicate `uv pip install` and duplicate log output.

### Expected behavior after fix

Before:
```
-> cd /path/to/worktree
direnv: loading .envrc
direnv: loading .erk/bin/activate.sh
-> cd /path/to/worktree          # from activate.sh (via direnv)
-> Activating venv: ...          # from activate.sh (via direnv)
Activated: /path/to/worktree
direnv: export +VIRTUAL_ENV ...
-> Activating venv: ...          # DUPLICATE from temp script
gt submit --no-interactive       # post_cd_commands
Checked out PR #8548 at ...
```

After:
```
-> cd /path/to/worktree
direnv: loading .envrc
direnv: loading .erk/bin/activate.sh
-> cd /path/to/worktree          # from activate.sh (via direnv)
-> Activating venv: ...          # from activate.sh (via direnv)
Activated: /path/to/worktree
direnv: export +VIRTUAL_ENV ...
                                 # guard skips ‚Äî VIRTUAL_ENV already set
gt submit --no-interactive       # post_cd_commands still run
Checked out PR #8548 at ...
```
